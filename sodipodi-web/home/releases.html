Sodipodi 0.34 release notes

This release incorporates for the first time path composition operations (union, intersection and subtraction) and new, calligraphic pen based freehand drawing. Plus many bugfixes and smaller features, as usual.

  • Fixed file and font name handling in Win9X/ME (Lauris)
  • Use full bounding box for calculating export region (John)
  • Support xml:space attribute for text (Mitsuru)
  • Support CDATA nodes (Lauris)
  • Preserve arbitrary namespace declarations (Lauris)
  • Path uncross, union, intersection and subtraction (Lauris)
  • Reimplemented printing from command line (Emile)
  • Bugfixes (everyone)

Sodipodi 0.29 release notes

This is the first version using Gtk+ 2.0 widget set and updated support libraries. So it is probably quite buggy inside and out.

The number of required libraries has come down considerably. Most importantly you do not need gnome libraries anymore, libgnomeprint and libgnomeprintui are optional. Thus building it should be much easier.

This version should compile on Windows 95/98/ME/2000/XP. The support is still in very early phase of development, so it may require some tweaking, depending on actual build system used.

Highlights (in the order of Changelog):

  • Port to Gtk+ 2.0 (Mitsuru)
  • Removed libgnome, libgnomeui, libgal dependencies, reimplement missing pieces (Mitsuru)
  • Gnome print support is now compile time option (Lauris)
  • Smooth canvas scroll (Lauris)
  • Initial direct PostScript print implementation (Lauris)
  • Windows32 port (Lauris)
  • Support GtkIM in text context (MenTaLguY)
  • Fixed namespace loading code (MenTaLguY)
  • Optional loading of Xft font database (Lauris)
  • Control constrains draw to 15 degrees instead of 90 (Nathan)
  • Fixed linear grid snapping code (Nathan)
  • Allow multiple toolboxes and toolbox docking (Lauris)

Sodipodi 0.25 release notes

Too much time has passed from the last release. Lot has changed inside sodipodi, so do expect this release to be buggier and slower than 0.24. But otherwise it is cool and no doubt together we can make it better for the next release.

Highlights:

  • Text object manages now the whole tree of <tspan> and TEXT nodes, multiline text thus being compatible with SVG specification. Old sodipodi files will be converted to new format automatically if opened.
  • Linear gradient unit space and transformation is now calculated correctly. Only viewport relative percentages are not supported.
  • Style parsing and inheritance is much better now. Specifically most distances are now handled correctly.
  • There is working pen tool in addition to pencil.
  • XML tree is now completely editable. (Pawel)
  • Vertical writing mode is supported, although font metrics are emulated.
  • Line width behaves as it should bot in display and print.
  • Text object gradient is correct in display mode too.
  • Stroke can have gradient fill. No printing yet, but it does work in display and bitmap export.
  • Command line arguments are working. Plus there are arguments to convert SVG to PNG from command line (no X needed)
  • There are nifty star, spiral and arc tool, the latter replacing old ellipse tool. (Mitsuru)
  • Bitmaps can be traced to vector from context menu, using autotrace and frontline. (Masatake)
  • Several objects have now property dialogs and node editing mode. (Mitsuru)
  • There is tool options dialog for some tools.

Sodipodi History

At the beginning there was Gill; this was meant to be the Gnome illustration program. Gnome initially had big hopes around the DOM infrastructure. I started playing with Gill and ezpaint code. EZPaint was an experimental set of interactive items for gnome-canvas.

It took about a year before people got used of sodipodi as The Gnome Illustaration program, then about a year ago Mitsuru or somebody asked, that I should try to port it to Windows. This opened my eyes. Ah, and I got letter from some people. Someone said, he was using KDE, he disliked gnome, but he installed it just to compile sodipodi :-)

So basically sodipodi has just been evolving, making some twists here and there, and nobody knows where it will evolve in next 2 years.

I like the fact that it still has potential for more than one application unlike gnumeric, for example which was, is and will be spreadsheet very probably.

-- Lauris, 2003
